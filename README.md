# Web Server Log Analyzer

A Python tool that parses Apache/Nginx access logs, flags suspicious activity
(brute-force login attempts, page/directory scanning, and request-rate abuse),
and generates a summary report in both plain-text and HTML formats.

This project was built as a digital forensics / SOC analyst portfolio piece —
it demonstrates the kind of log-triage workflow a junior security analyst or
sysadmin does every day, just automated.

---

## What log files actually look like

Before writing any parsing code it's worth understanding the three log types
this project touches on:

**Apache / Nginx access logs (`access.log`)** — one line per HTTP request, in
the "combined log format":

```
127.0.0.1 - - [10/Oct/2023:13:55:36 -0700] "GET /index.html HTTP/1.1" 200 2326 "-" "Mozilla/5.0"
```

Fields: client IP, remote user (usually `-`), authenticated user (usually
`-`), timestamp, HTTP method + requested URL + protocol, status code,
response size in bytes, referrer, and user-agent string.

**Linux `auth.log` / `secure`** — records authentication events (SSH logins,
`sudo` use, failed password attempts) on Debian/Ubuntu (`/var/log/auth.log`)
and RHEL/CentOS (`/var/log/secure`). A failed SSH login looks like:

```
Jun 15 08:03:11 web01 sshd[2211]: Failed password for root from 203.0.113.66 port 51422 ssh2
```

**Windows Event Logs** — stored in a binary `.evtx` format and viewed through
Event Viewer or exported to CSV/XML. Security-relevant entries use Event IDs,
e.g. `4625` = failed logon, `4624` = successful logon, `4720` = new user
account created.

This project focuses on the first type (web server access logs), since
that's the most common log an entry-level analyst is asked to triage, and the
plain-text format makes it a good regex-parsing exercise. The detection
logic (rate limiting, brute force, scanning) is conceptually the same
pattern you'd apply to auth.log or exported Windows events — only the field
layout changes.

## What the tool does

1. **Parses** each log line with a regex, pulling out:
   - IP address
   - Timestamp
   - HTTP method (GET, POST, etc.)
   - Requested URL
   - HTTP status code
   - Response size

2. **Detects suspicious activity**:
   | Detection | Rule | Why it matters |
   |---|---|---|
   | Rate abuse | An IP makes more than 50 requests in a single minute | Could be a bot, scraper, or denial-of-service attempt |
   | Brute force | An IP generates 10+ `401 Unauthorized` responses | Someone repeatedly guessing login credentials |
   | Scanning | An IP generates 15+ `404 Not Found` responses | Someone probing for hidden admin pages, config files, or known vulnerable paths |

   These thresholds are constants at the top of `log_analyzer.py`
   (`RATE_LIMIT_PER_MINUTE`, `BRUTE_FORCE_401_THRESHOLD`,
   `SCANNING_404_THRESHOLD`) and can be tuned for a busier or quieter site.

3. **Summarises** the traffic:
   - Top 10 requesting IPs
   - Top 10 most-visited URLs
   - Status code breakdown
   - Overall error rate (percentage of requests returning 4xx/5xx)
   - Full list of flagged IPs with the reason(s) they were flagged

4. **Reports** the summary as:
   - A plain-text file (`*_report.txt`) — easy to grep, email, or paste into
     a ticket
   - An HTML file (`*_report.html`) — a formatted table view, easy to open
     in a browser or attach to an incident write-up

   Both reports are timestamped with the exact time they were generated.

## Usage

```bash
# Analyse a single log file
python3 log_analyzer.py sample_logs/access_log_2_bruteforce.log

# Analyse several files at once (e.g. logs rotated daily) into one summary
python3 log_analyzer.py sample_logs/*.log --outdir reports --prefix weekly_summary
```

Reports are written to the `reports/` folder (or wherever `--outdir` points)
as `<prefix>_report.txt` and `<prefix>_report.html`.

No third-party packages are required — only the Python standard library
(`re`, `collections`, `datetime`, `html`, `argparse`).

## Test log files

Four synthetic log files are included in `sample_logs/` (generated by
`generate_sample_logs.py`, all fake IPs from documentation/test ranges —
no real user data):

| File | Scenario | Expected result |
|---|---|---|
| `access_log_1_normal.log` | Ordinary browsing traffic only | 0 IPs flagged |
| `access_log_2_bruteforce.log` | Normal traffic + one IP hammering `/login` with 401s | 1 IP flagged (brute force) |
| `access_log_3_scanning.log` | Normal traffic + one IP probing admin/config paths | 1 IP flagged (scanning) |
| `access_log_4_rateflood.log` | Normal traffic + one IP sending 70 requests in one minute | 1 IP flagged (rate abuse) |

Running the analyzer against all four files together correctly flags all
three attacker IPs with zero false positives against the legitimate traffic.

To regenerate the sample logs: `python3 generate_sample_logs.py`

## Why this matters — what a SOC analyst / sysadmin actually uses this for

Server logs are one of the first things a **Security Operations Center (SOC)
analyst** or **sysadmin** looks at when something seems off — a slow server,
a defaced page, an unexpected AWS bill, or a tip-off from a customer. Log
analysis is central to several parts of the job:

- **Incident detection & triage.** Most attacks leave a trail in the access
  logs before anything else notices — repeated failed logins, a burst of
  requests to `/wp-admin` or `/.env`, or a single IP suddenly generating
  thousands of requests. Catching this early (ideally automatically) is
  often the difference between stopping an attack and cleaning up after a
  breach.

- **Brute-force and credential-stuffing detection.** A wall of `401`
  responses from one IP against a login endpoint is a classic sign of
  someone trying to guess passwords or replay leaked credential lists. This
  is one of the most common alerts a SIEM (Security Information and Event
  Management) tool generates, and understanding the underlying log pattern
  is essential to know when the alert is real versus noise.

- **Reconnaissance / vulnerability scanning detection.** Attackers (and
  automated bots) routinely scan sites for exposed admin panels, `.git`
  folders, backup files, and known-vulnerable scripts before attempting an
  actual exploit. A spike in `404` responses from one source is often the
  earliest warning that a target has been noticed.

- **Capacity planning and abuse prevention.** Not every anomaly is
  malicious — a sudden spike in traffic might just be a marketing campaign
  going well. But distinguishing "we went viral" from "we're being scraped
  or DDoS'd" starts with the same rate-analysis techniques used here.

- **Compliance and post-incident forensics.** After a breach, investigators
  reconstruct what happened using logs: which endpoints were hit, from
  where, and when. Having clean, summarised, timestamped reports (like the
  ones this tool produces) makes that process faster and gives auditors or
  management something readable instead of raw log dumps.

- **Building blocklists and firewall rules.** The output of this kind of
  analysis (a list of flagged IPs with reasons) feeds directly into actions
  like adding IPs to a WAF (Web Application Firewall) blocklist, `fail2ban`
  rule, or SIEM watchlist.

In a real SOC this same pipeline exists at a much larger scale — tools like
**Splunk, ELK/Elastic Stack, Graylog, or Microsoft Sentinel** ingest millions
of log lines per day, run detection rules very similar in concept to the
ones in this project (thresholds, pattern matching, correlation across
sources), and raise alerts for a human analyst to investigate. Writing this
tool from scratch is a good way to understand what those platforms are
actually doing under the hood, rather than just clicking through a
pre-built dashboard.

## Possible extensions

- Parse Linux `auth.log` in addition to web logs (different regex, same
  detection framework)
- Support Windows Event Log CSV exports (Event ID 4625 = failed logon)
- Cross-reference flagged IPs against a threat-intel feed or GeoIP lookup
- Auto-generate `fail2ban` or firewall block rules from the flagged IP list
- Add a rolling/live mode that tails a log file in real time
